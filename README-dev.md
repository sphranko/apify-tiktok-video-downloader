## Architecture overview

```
profileUrls (input)
       │
       ▼
  scraper.js          ← yt-dlp --dump-json (no download, JSONL output)
       │  normalizes to flat dataset record
       ▼
  main.js             ← orchestration loop
       │
       ├──[downloadVideos=true]──► downloader.js   ← yt-dlp -o /tmp/...mp4
       │                                │ reads buffer, stores to KV store
       │                                ▼
       │                          KeyValueStore  (video-<id>.mp4)
       │
       └──────────────────────────► Dataset       (one record per video)
```

The actor has no browser dependency. All TikTok interaction goes through **yt-dlp**, which handles API signing, CDN authentication, and pagination internally.

## Data flow

### 1. Input

`main.js` reads the actor input via `Actor.getInput()`:

| Field            | Type       | Default  |
|------------------|------------|----------|
| `profileUrls`    | `string[]` | required |
| `limit`          | `integer`  | `10`     |
| `order`          | `string`   | `"desc"` |
| `downloadVideos` | `boolean`  | `true`   |

### 2. Scraping (`scraper.js`)

```
yt-dlp --dump-json --no-download --playlist-items 1:N <profileUrl>
```

- `N` = `limit` for `desc` order; `limit × 5` (max 100) for `asc` order
  (TikTok always returns newest-first, so for ascending order we fetch a larger
  window, then sort and slice)
- Output: JSONL (one JSON object per line, one per video)
- Parsed, sorted by `timestamp`, normalised, filtered to only the target user's
  content (yt-dlp can return reposts or recommended content in some responses)
- Returns `{ videos: Object[] }` sliced to `limit`

### 3. Download (`downloader.js`)

```
yt-dlp -o /tmp/tiktok-dl/video-<id>.mp4 --format best[ext=mp4]/best <videoPageUrl>
```

- Temp file written to `/tmp/tiktok-dl/` (guaranteed writable in both Apify and
  local environments)
- File read into a `Buffer`, temp file deleted, buffer returned
- Caller (`main.js`) stores the buffer in the KV store with `contentType: 'video/mp4'`

### 4. Dataset record

Each record pushed to the dataset is a flat object. Fields that only appear when
`downloadVideos` is enabled and the download succeeded: `storageKey`, `storageUrl`.

## How yt-dlp is used

### Why yt-dlp?

TikTok's API requires a cryptographic signature (`X-Bogus`, `_signature`) that is
generated by a proprietary JavaScript/Java library embedded in the TikTok web app.
Replicating this from scratch is fragile and frequently breaks.

yt-dlp bundles its own TikTok extractor that reverse-engineers this signing process
and keeps it updated. This is the same approach used by the `dltik` Apify actor.

The `clockworks/tiktok-scraper` actor achieves the same by running the real TikTok
MetaSec Java signing library inside a CheerioCrawler setup — an approach that
requires a JVM and the proprietary `.jar`, which we deliberately avoid.

### Key yt-dlp flags

| Flag | Purpose |
|------|---------|
| `--dump-json` | Print full info dict for each video; do not download |
| `--no-download` | Safety — paired with `--dump-json` |
| `--no-warnings` | Suppress stderr noise |
| `--no-check-certificates` | Required in some Apify network environments |
| `--playlist-items 1:N` | Limit how many videos are fetched from the profile |
| `--format best[ext=mp4]/best` | Prefer native MP4 container when downloading |

### yt-dlp JSONL → normalised record

`scraper.js` maps yt-dlp's raw info dict fields to the actor's output shape,
which is modelled after `clockworks/tiktok-scraper` for drop-in compatibility:

| yt-dlp field | Actor field |
|---|---|
| `id` | `id` |
| `description` / `title` | `text` |
| `timestamp` | `createTimeISO` (ISO string) |
| `uploader` / `uploader_id` | `author.name` |
| `channel_url` | `author.avatar` |
| `view_count` | `playCount` |
| `like_count` | `diggCount` |
| `comment_count` | `commentCount` |
| `repost_count` | `shareCount` |
| `duration` | `video.duration` |
| `thumbnail` | `video.cover` |
| `width` / `height` | `video.width` / `video.height` |
| `track` | `music.name` |
| `artist` | `music.author` |
| `webpage_url` | `webVideoUrl` |

## Output data shape

Stats are flat top-level keys; author, video, and music info are nested objects.
The Apify dataset view references nested fields using dot paths (e.g. `video.cover`).

```jsonc
{
  "id": "7380123456789012345",
  "text": "Video caption here",
  "createTimeISO": "2024-04-25T10:30:00.000Z",

  "author": {
    "name":   "tiktok",
    "avatar": "https://www.tiktok.com/@tiktok"   // channel URL, not image
  },

  "playCount":    1234567,
  "diggCount":    54321,
  "commentCount": 987,
  "shareCount":   123,
  "collectCount": 0,          // always 0 — yt-dlp does not expose this

  "video": {
    "duration": 15,           // seconds
    "cover":    "https://p16-sign.tiktokcdn-us.com/...",
    "width":    576,
    "height":   1024
  },

  "music": {
    "name":     "Original Sound",
    "author":   "tiktok",
    "original": true
  },

  "webVideoUrl": "https://www.tiktok.com/@tiktok/video/7380123456789012345",

  // Only present when downloadVideos=true and download succeeded:
  "storageKey": "video-7380123456789012345.mp4",
  "storageUrl": "https://api.apify.com/v2/key-value-stores/<storeId>/records/video-7380123456789012345.mp4"
}
```

> **Note:** `collectCount` is always `0` because yt-dlp does not return the
> TikTok "Saved" (bookmark) count. `music.*` fields may be empty strings if
> yt-dlp does not identify the track.

## Apify storage layout

### Dataset
One record per video (see shape above). Accessible via the run's default dataset.

### Key-Value Store
MP4 files stored with key pattern `video-<videoId>.mp4` and content type `video/mp4`.

### `storageUrl` construction

The KV store public URL is hardcoded to `https://api.apify.com` (not
`process.env.APIFY_API_BASE_URL`) because on Apify infrastructure that env var
resolves to an internal IP (`http://10.x.x.x`), which is not accessible externally.

```js
const APIFY_API_BASE = 'https://api.apify.com';
record.storageUrl = `${APIFY_API_BASE}/v2/key-value-stores/${store.id}/records/${storageKey}`;
```

## Local development

### Prerequisites

- Node.js ≥ 20
- [yt-dlp](https://github.com/yt-dlp/yt-dlp) installed and on `$PATH`
- [Apify CLI](https://docs.apify.com/cli/) — `npm install -g apify-cli`

### Setup

```bash
git clone https://github.com/sphranko/apify-tiktok-video-downloader.git
cd apify-tiktok-video-downloader
npm install
```

### Input file

`.actor/input.json` is already set up. Edit it to change the test profile:

```json
{
    "profileUrls": ["https://www.tiktok.com/@tiktok"],
    "limit": 3,
    "order": "desc",
    "downloadVideos": false
}
```

### Run

```bash
apify run          # uses .actor/input.json automatically
```

Dataset results appear in `storage/datasets/default/`.
Downloaded MP4s appear in `storage/key_value_stores/default/`.

### Run without Apify CLI

```bash
node src/main.js
```

This requires `APIFY_TOKEN` and a valid `INPUT` key in the default KV store,
or the actor will throw on missing input. Prefer `apify run` for local testing.

## Docker image

Base image: `apify/actor-node:20` (Alpine Linux + Node.js 20 LTS).

yt-dlp is installed at Docker **build time** (not runtime) to keep startup fast:

```dockerfile
RUN apk add --no-cache python3 && \
    python3 -c "import urllib.request; urllib.request.urlretrieve( \
        'https://github.com/yt-dlp/yt-dlp/releases/latest/download/yt-dlp', \
        '/usr/local/bin/yt-dlp')" && \
    chmod +x /usr/local/bin/yt-dlp
```

> **Heads-up:** `apify/actor-node:20` is Alpine-based — use `apk`, not `apt-get`.

## Known limitations

| Limitation | Detail |
|---|---|
| `collectCount` always 0 | TikTok's "saved" count is not exposed by yt-dlp |
| `asc` order is approximate | TikTok returns videos newest-first; we fetch up to `limit × 5` (max 100), then sort and slice — oldest videos beyond 100 are unreachable |
| CDN URL expiry | TikTok CDN URLs are time-limited; the actor downloads immediately after scraping |
| Rate limiting | Running against many profiles rapidly may trigger TikTok throttling |
| yt-dlp version drift | yt-dlp is fetched at Docker build time; rebuild the image to pick up TikTok extractor fixes |
| No repost support | Reposts from other accounts are filtered out (author name check in `scraper.js`) |

## Design decisions & history

| Decision | Reason |
|---|---|
| yt-dlp instead of Playwright | Playwright was detected as a bot by TikTok and caused memory issues (> 976 MB in a 1 GB container). yt-dlp works from datacenter IPs without a browser. |
| yt-dlp instead of CheerioCrawler + SSR | TikTok's SSR payload does not include the video list when served to datacenter IPs — only profile metadata. |
| No axios / node-fetch dependency | yt-dlp handles all HTTP, including CDN auth. Only `apify` and `crawlee` (for logging) are needed. |
| Flat dotted keys (`authorMeta.name`) | Apify dataset views require flat key names; nested objects cannot be referenced in `transformation.fields`. |
| Hardcoded `https://api.apify.com` | `APIFY_API_BASE_URL` resolves to an internal IP on Apify infra, producing non-public `storageUrl` values. |
| `collectCount: 0` kept in output | Maintains schema compatibility with `clockworks/tiktok-scraper` for consumers that expect the field. |
